# Problem 03: The Hybrid Approach — X25519Kyber768 in TLS 1.3

## Problem Statement

A security engineer is deploying TLS 1.3 on a high-traffic API gateway serving 50,000
connections per second. They want to adopt post-quantum key exchange while maintaining
backward compatibility with legacy clients.

1. Design the TLS 1.3 extension structure for X25519Kyber768 hybrid key exchange.
   Specify the exact bytes of the `key_share` extension for both ClientHello and
   ServerHello.

2. Derive the combined shared secret from both the classical X25519 and PQ Kyber-768
   components. Show how the combined secret enters the TLS 1.3 key schedule.

3. Analyse the performance impact: key generation time, encapsulation/decapsulation
   time, and handshake latency overhead at 50,000 connections/second.

4. A legacy client only supports X25519. Show what happens during the TLS negotiation
   when the server offers both X25519 and X25519Kyber768.

---

## Solution

### Part 1 — Extension Structure

**NamedGroup identifier for X25519Kyber768:**

The IETF hybrid key exchange draft defines:
```
X25519Kyber768Draft00 = 0x6399   (draft codepoint; final RFC will assign permanent value)
```

**ClientHello `supported_groups` extension:**

```
Extension Type:     0x000A  (supported_groups)
Extension Length:   0x0008  (8 bytes)
Group List Length:  0x0006
Group List:
  0x6399    X25519Kyber768Draft00   (hybrid PQ+classical)
  0x001D    X25519                  (classical fallback)
  0x0017    P-256                   (additional classical fallback)
```

**ClientHello `key_share` extension:**

The client sends a key share for its preferred group (X25519Kyber768):

```
Extension Type:     0x0033  (key_share)
Extension Length:   0x04C6  (1222 bytes total)
Client Shares Length: 0x04C4

Key Share Entry:
  Group:    0x6399   (X25519Kyber768)
  Key Exchange Length: 0x04C0  (1216 bytes)
  Key Exchange Data:
    Bytes 0..31:      X25519 client public key (32 bytes)
    Bytes 32..1215:   Kyber-768 client public key (1184 bytes)
    Total:            1216 bytes
```

The hybrid key share is the **concatenation** of the classical and PQ public keys
in the `key_exchange` field. Both components are generated by the client independently
and concatenated in a fixed order (X25519 first, then Kyber).

**ServerHello `key_share` extension:**

The server responds with its X25519 public key and the Kyber ciphertext:

```
Extension Type:     0x0033  (key_share)
Extension Length:   0x0467  (1127 bytes total)
Key Share Entry:
  Group:    0x6399   (X25519Kyber768)
  Key Exchange Length: 0x0460  (1120 bytes)
  Key Exchange Data:
    Bytes 0..31:      X25519 server public key (32 bytes)
    Bytes 32..1119:   Kyber-768 encapsulation ciphertext (1088 bytes)
    Total:            1120 bytes
```

The server:
1. Generates a fresh X25519 key pair (ephemeral).
2. Runs Kyber-768 `Encapsulate` on the client's Kyber-768 public key, producing:
   - A 1088-byte ciphertext (sent to client)
   - A 32-byte Kyber shared secret (kept internally)

### Part 2 — Shared Secret Derivation

**Client side:**

```
// Step 1: X25519 shared secret
x25519_shared = X25519(client_x25519_private, server_x25519_public)
             -- 32 bytes

// Step 2: Kyber shared secret
kyber_shared = Kyber768_Decapsulate(client_kyber_private, server_kyber_ciphertext)
            -- 32 bytes

// Step 3: Combine
DHE_combined = x25519_shared || kyber_shared
            -- 64 bytes (concatenation)
```

**Server side:**

```
// Step 1: X25519 shared secret (identical computation, commutativity)
x25519_shared = X25519(server_x25519_private, client_x25519_public)
             -- 32 bytes (same value as client's)

// Step 2: Kyber shared secret (server already computed this during Encapsulate)
kyber_shared = K_encapsulate  -- the Kyber shared secret from Encapsulate step
            -- 32 bytes (same value as client's)

// Step 3: Combine
DHE_combined = x25519_shared || kyber_shared
            -- 64 bytes
```

**Entry into TLS 1.3 key schedule:**

The 64-byte `DHE_combined` replaces the standard 32-byte X25519 output as the IKM
(input key material) in HKDF-Extract for the Handshake Secret:

```
# Standard TLS 1.3 (X25519 only):
Handshake_Secret = HKDF-Extract(
    salt = Derive-Secret(Early_Secret, "derived", ""),
    IKM  = X25519_shared   -- 32 bytes
)

# Hybrid X25519Kyber768:
Handshake_Secret = HKDF-Extract(
    salt = Derive-Secret(Early_Secret, "derived", ""),
    IKM  = DHE_combined    -- 64 bytes (X25519_shared || Kyber_shared)
)
```

The HKDF-Extract function accepts variable-length IKM, so no other changes to the
TLS 1.3 key schedule are required. All subsequent derivations (handshake traffic
secrets, master secret, application traffic secrets) proceed identically to standard
TLS 1.3.

**Security argument for the combination:**

The security of $\text{HKDF-Extract}(\text{salt}, X\|K)$ where $X$ is the X25519
secret and $K$ is the Kyber secret follows from the HKDF PRF property:

- If X is computationally indistinguishable from random (X25519 security): the output
  is secure even if K is adversarially controlled.
- If K is computationally indistinguishable from random (ML-KEM security): the output
  is secure even if X is adversarially controlled.

The concatenation is secure as long as at least one component is computationally
indistinguishable from random — this is the formal security guarantee of the hybrid.

### Part 3 — Performance Analysis at 50,000 Connections/Second

**Per-operation costs (measured on a modern server, x86-64, 2024):**

```
Operation                    Time (approx)    Notes
---------------------------  -------------    -------------------------
X25519 key generation         5 µs            Scalar mult on Curve25519
X25519 shared secret          5 µs            Same cost as key gen
Kyber-768 KeyGen             30 µs            NTT-based; Intel AVX2
Kyber-768 Encapsulate        45 µs            NTT-based; Intel AVX2
Kyber-768 Decapsulate        55 µs            NTT-based; includes FO re-encrypt
```

**Server-side operations per TLS connection (X25519Kyber768):**

```
Client operations:
  X25519 key gen:      5 µs
  Kyber-768 key gen:  30 µs
  Total client:       35 µs per connection

Server operations (in order):
  X25519 key gen:      5 µs   (ephemeral)
  Kyber-768 Encap:    45 µs   (using client's Kyber public key)
  X25519 shared:       5 µs   (using client's X25519 public key)
  Total server:       55 µs per connection
  (+ AES-GCM, certificate ops, etc., which are unchanged)
```

**Comparison to pure X25519:**

```
Pure X25519 server ops:   5 + 5 = 10 µs
Hybrid X25519Kyber768:   55 µs
Overhead factor:         5.5× slower per connection
```

**At 50,000 connections/second:**

```
Pure X25519 CPU time:      50,000 × 10 µs = 500,000 µs = 0.5 CPU-seconds
X25519Kyber768 CPU time:   50,000 × 55 µs = 2,750,000 µs = 2.75 CPU-seconds

Additional CPU requirement: 2.25 CPU-seconds per second = 2.25 cores
```

With AVX-512 acceleration (and Kyber AVX-512 assembly):
```
Kyber-768 Encap (AVX-512): ~15 µs
Server ops with AVX-512:    5 + 15 + 5 = 25 µs
CPU at 50k conns/sec:       50,000 × 25 µs = 1.25 CPU-seconds
Additional cores needed:    ~0.75 cores over pure X25519
```

**Network bandwidth overhead:**

```
Additional bytes per handshake (X25519Kyber768 vs X25519):
  ClientHello key_share:   +1184 bytes (Kyber pk)
  ServerHello key_share:   +1088 bytes (Kyber ct)
  Total:                   +2272 bytes per TLS session

At 50,000 connections/second:
  Additional bandwidth:    50,000 × 2272 / 1,000,000 = 113.6 Mbps
```

On a 10 Gbps link, this is 1.1% bandwidth overhead — negligible.

**Conclusion:** The hybrid approach is operationally feasible at 50,000 connections/sec.
A modest increase in CPU usage (approximately 2-3 additional cores) handles the Kyber
overhead with standard library implementations. With AVX-512 hardware acceleration,
the overhead is under 1 core. The bandwidth overhead is below 1.2% on typical links.

### Part 4 — Legacy Client Negotiation

**Scenario:** Legacy client supports only X25519 (not X25519Kyber768).

**Client's extensions:**

```
supported_groups: [X25519, P-256]   (no hybrid group listed)
key_share:        [(X25519, client_x25519_pk)]   (only X25519 share)
```

**Server behaviour:**

The server iterates through the client's `key_share` entries looking for a group it
also supports. The server supports X25519Kyber768 and X25519. Since the client's
`key_share` contains only X25519 (not X25519Kyber768), the server:

**Option A — Fallback to X25519 (normal path):**

If X25519 appears in both the client's `supported_groups` and `key_share`, and the
server also supports X25519, the server responds with:

```
ServerHello:
  key_share: (X25519, server_x25519_pk)   -- pure classical X25519
```

The handshake completes as a standard TLS 1.3 connection with X25519 key exchange —
no hybrid, no post-quantum security, no forward PQ secrecy. The session is HNDL-vulnerable.

**Option B — HelloRetryRequest (if X25519 not in key_share but in supported_groups):**

If the client listed X25519 in `supported_groups` but did not include a key share for
it (and the server cannot use X25519Kyber768 because the client doesn't support it),
the server sends a HelloRetryRequest specifying X25519. The client then resends
ClientHello with an X25519 key share.

**Security analysis of the fallback:**

The fallback to X25519 means:
- The session has no quantum-resistant forward secrecy.
- HNDL attacks are possible for the session's forward secrecy.
- Authentication (via TLS certificate) may or may not be PQ-protected depending on
  whether the server's certificate uses ML-DSA or ECDSA.

**Transparency and policy:**

The server cannot force a PQ-capable client to use X25519Kyber768 — the client's
`supported_groups` determines what is acceptable. A deployment policy requiring
PQ key exchange can be enforced by:

1. Rejecting connections from clients that don't offer X25519Kyber768 (not backward
   compatible — breaks legacy clients).
2. Logging connections that fall back to X25519 (for monitoring migration progress).
3. Accepting X25519 fallback but documenting that HNDL protection requires clients
   to be updated.

**Recommended deployment approach:**

```
Phase 1 (now):    Accept both X25519Kyber768 and X25519; log X25519-only connections
Phase 2 (2026+):  Alert operators when X25519-only fraction exceeds threshold
Phase 3 (2028+):  Require X25519Kyber768; reject X25519-only clients for high-security APIs
```

This matches how the industry transitioned from SSL/TLS 1.0 to TLS 1.2 and TLS 1.3:
a gradual deprecation of older protocols with monitoring and eventual enforcement.
